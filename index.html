​ 在 AI 的帮助下，阿里巴巴闲鱼技术团队正在推进一种智能工具，这种工具可以将视觉 UI 设计转换为具有 100% 准确性的 Flutter 代码。

​ 正如我们在最近关于基于 AI 的算法开发的文章中所探讨的那样，自动化技术正在迅速地从需要开发者来进行工作的东西转变为不需要开发者就可以工作的东西。 虽然这种转变的程度在未来几年仍有待观察，但今天的一个主要焦点是 AI 如何解决应用程序开发中的重复和乏味，以便人类开发者可以将他们的工作重点放在其他地方。

​ 现在，阿里巴巴闲鱼二手交易平台的开发者推出了 UI2CODE 项目，将深度学习技术应用于将可视化用户界面图像转换为客户端代码。 凭借清晰的组件，位置和适合机器学习的布局特征，UI 视觉研究为探索这些技术在自动代码生成中的应用提供了一个特别有前景的领域。

​ 在介绍闲鱼正在进行的工作中，我们将探讨 UI2CODE 如何分析 GUI 元素和 UI 布局结构，以便为应用程序的整体设计生成 Flutter 代码。

Fundamentals of the Project | 项目基础

​ UI2CODE 的早期工作始于 2018 年 3 月，当时闲鱼技术团队对其技术可行性进行了初步研究。 从那时起，该项目经历了三轮重构，通过各种机器学习代码生成的程序来进行，以满足商业环境的标准。

​ UI2CODE 的核心原则是划分和控制 UI 开发功能，以避免太多鸡蛋在一篮子的情况。 为此，它所包含的解决方案已根据三个核心需求进行了评估。 首先，对于视觉修复的精确度，它们的输出不会偏离甚至单个像素。 其次，虽然机器学习受概率影响，但其结果一定可以达到 100％ 的准确度标准。 最后，解决方案必须易于维护; 对于工程师来说，可理解和可修改只是一个起点，而且合理的布局结构对于确保平滑的界面操作来说至关重要。

Component Identification 组件识别

​ 在识别组件之前，必须收集样品组件进行培训。 此外，TensorFlow 中提供的 CNN 模型和 SSD 模型在此阶段也用于增量训练。

​ UI2CODE 根据各种类型对 GUI 元素进行分类，包括图像，文本，形状/按钮，图标，价格等，然后将其分类为 UI 组件，CI 组件和 BI 组件。 UI 组件主要用于 Flutter 本机组件的分类; CI 组件主要用于闲鱼的定制 UIkits 的分类; BI 组件主要用于对具有特定业务相关性的源流卡进行分类。

Layout Analysis | 布局分析

​ 在 UI2CODE 的早期阶段，闲鱼团队使用四层 LSTM 网络进行培训和学习，然后由于样本量较小而转向规则实施。 规则实施还具有相对简单的优点; 切片第一步的五个切口的顺序是 row 和 col。 规则实施的缺点是布局相对严格，需要与 RNN 结合以提前反馈。

Code Generation and Plug-in 代码和生成插件

​ AI 基本上是概率问题，而自动生成的代码需要非常高的可恢复性和 100％ 的准确性。并且由于概率的浮动使得 AI 难以达到这种精确度，因此需要一个可编辑的工具来使开发人员能够快速理解和修改 UI 布局结构。

​ 为此，闲鱼团队为 UI2CODE 生成的 DSL TREE 实现了基于模板的匹配方法，其中代码模板的内容由经验丰富的 Flutter 技术人员定义。 到目前为止，这已被证明是代码实现的最佳方法。

​ 代码模板中被引入了一些标记，Intellij 插件用于检索和替换 Flutter 项目中的相应 UIkit，以便提高代码重用率。

​ 整个插件项目需要提供自定义 UIkit 功能，包括 DSL 树的检索，替换，验证，创建，修改和图形识别。 总之，这与 ERP 系统类似，后者则需要时间进行改进。

Key Takeaways 关键要点

​ 在本文介绍的五个关键组件中，有四个涉及机器视觉问题，并通过 AI 连接在一起。对将代码发布到在线环境中这件事提出了非常严格的要求，并且对 AI 的概率性质这方面提出了重大挑战。 为了继续解决这些问题并实施解决方案，闲鱼技术团队选择专注于机器视觉功能，同时使用 AI 技术作为整体构建 UI2CODE 系统的补充，并继续专注于 AI 技术，使 UI2CODE 成为自动化代码生成工具的一种理想选择。

Aboutface4交互设计精髓 Chapter1 笔记

发表于 2019-03-10 | 更新于 2019-03-14

Chapter1. 数字产品的设计过程

一个前提：只要用户满意，产品就会成功。
一个问题：在产品规划和开发的过程中，我们没有把设计作为同等重要的基础环节。

我们将 设计 更加具体地定义为一种以人为本的设计活动。

  • 理解用户的期望、需求、动机和使用情景。
  • 理解商业、技术以及行业的机会、需求和制约。

运用恰当的方法，设计能够填上人类与科技产品之间的缺口。
在这里，我认为设计的本质就是平衡，平衡用户体验与商业利益，平衡生活与科技。

产品行为恶劣的后果

  • 数字产品要求人们像计算机一样思考
    数字产品经常假设人们非常了解技术，因而常常晦涩难懂，把真实含义、意图和动作隐藏起来，让用户无从知晓。
    其实就是我们有时会依照实现模型去设计产品，而不是依照用户的心理模型。
  • 数字产品马虎大意
    软件常常要求人们脱离主任务流，执行一些本不需要另外打开一个界面或额外导航才能完成的功能。
    例如，DropBox 的上下文菜单中，删除键位于下载和重命名之间，非常容易让人不小心误删原本为安全起见上传到云端的文件。
    在这里，我们可以用 “序列效应” 来改进功能按键的排序，

此外，商业和科技应用程序的外观复杂混乱，使导航和理解变得困难，这完全没有必要。

  • 数字产品要求人来干重活
    仅仅为了恰当的地操作软件，人们被迫做大量的工作。

为何数字产品表现如此糟糕？

  • 重点错置
    产品管理和开发团队工作重点错置。
    一个数字产品经常容易受到 市场营销人员 与 开发人员 这两个对立阵营的影响。
  1. 市场营销人员 擅长理解和估计商机，精于向市场推出和定位产品，但他们对产品设计过程中的贡献通常仅限于一点需求列表而已。这些需求通常和期望关系不大，更多的是为了追逐竞争、用任务列表管理 IT 资源。
    （很少有用户能够清晰地描述自己的需求，直接问他们使用产品的感受，大多都倾向于关注产品的次要功能，或者弥补缺陷的小窍门，或者用户认为他们要买什么样的产品，并没有透露他们如何或者是否会使用这种产品。）
  2. 开发人员 负责产品的构造和过程，但是开发人员的任务诉求与产品最终用户完全不同。
    优秀的开发人员关注的是解决技术难题带来的挑战、遵循优秀的工程实践标准，从而如期完成任务。
    他们收到的指示可能会不完整、缺乏远见、有时甚至互相矛盾。还要在极为紧迫的时间内，在并不了解用户如何使用产品的情况下，被迫做出关于用户体验的重要决定。

以上，因此，最直接负责创造数字产品的人员很少会考虑到用户的目标、需求或动机。他们更加关心市场潮流和突破技术限制
为了确保产品真正满足用户的需求，设计工作应该先于写代码和测试工作。

  • 无视产品的真实用户
    大部分技术产品都是在不太了解用户的情况下制造出来的。

  • 利益冲突
    它影响着产品制造商和经销商取悦用户的能力。
    开发者（程序员）需要在易于编程和易于使用之间做出选择，他们的业绩评估往往主要取决于其编程效率以及能否在极其紧张的时间内完成任务。
    程序员即使有足够的设计能力和设计意愿，也无法兼顾用户、商业和技术等各方利益。

  • 设计流程缺失
    遗漏了一个重要环节：
    即 确保产品能吸引用户的一个可重复的、可预见的分析过程
    这个环节可以把对用户的了解转化为满足用户专业化、个性化和情感化需求的数字产品。

现在许多公司认为让用户直接融入开发过程就能够解决人类交互设计方面的问题，这虽然能有效地让用户来分担设计责任，却忽略了方法上的一个严重不足：将设计知识同领域知识混为了一谈。

一个领域内的专家可能无法完整站在一个普通用户的立场上，去确定任务流。

对设计师和产品来说，听一听用户反馈的问题，远比直接从用户的得到解决方案更有价值。

规划并设计产品行为

有意识地引入设计，标志着现代产品开发三原则开始形成：

  • 功用性（capability）
  • 可行性（viability）
  • 称许性（desirability）
    这三元素的任何一个的弱化，都不可能使产品经受住时间的考验。

行为设计 需要更了解情景，而不仅仅是视觉组合和品牌规划。
行为设计 需要了解用户从购买到使用完整过程中同产品的关系。最重要的是，要了解用户希望如何使用该产品？ / 以什么样的方式使用产品？ / 使用产品的目的是什么？

交互设计并不仅仅是审美选择，更是要建立在用户和认知原则的了解上。

我们使用 目标导向设计 来了解用户目标、需求和动机
首先我们需要更好的了解用户目标的本质、产生用户目标的心理模型，以及用户目标如何成为设计交互行为的关键。

识别用户目标

只为实现商业目标而设计和开发的产品终会失败，因为它没有解决用户的个人需求。
当产品设计能够满足用户个人目标需要的时候，商业目标也会更有效地实现。

大多数商业软件、网站和数字产品存在的问题是：

  • 让用户感觉自己很愚蠢
  • 导致用户犯错太多
  • 要费很大力气才能有效操作
  • 无法带来愉快的体验
    开发这些产品的公司没有分清楚重点，多数只狭隘地关注产品的实际问题，分散了对用户需求的关注。

目标 vs. 任务和活动

目标不等于任务或活动（activity）。目标是对最终情况的预期，而任务和活动只是达成一个 或者一组目标的中间步骤（在组织中的不同层次上）。

唐纳德.诺曼（Donald Norman）提倡 “以活动为中心的设计”（Activity-Centered Design，ACD），强调首先要理解和重视活动。
他认为，理解人们如何适应手头的工具，以及使用工具展开的活动，就能更好地影响工具的设计。

诺曼的 ACD 方法强调用户情景的重要性
虽然 ACD 之类的方法对于恰当分解用户的行为 “是什么” 非常有效，但实际上并未解决每个设计者首先要提出的问题：用户一开始为什么要开展这个活动、任务、动作或者操作？
目标驱动人们开展活动，理解目标可以帮助你理解用户的预期和渴望，反过来又会帮你决定哪些行动的确和设计相关

目标、活动或任务之间的区别
由于目标受人类动机驱使，因而随时间的推移变化很慢，甚至没有变化。
行动和任务则易于变化，因为几乎完全依赖于手头技术水平。

设计要满足情景中的目标

设计目标依赖于具体的情景——用户是谁？在做什么？目标是什么？

优秀的设计让用户更有效率

虽然任务是用户的关注点，但设计者要关注任务之外的问题，明确最重要的用户是谁，进而确定用户目标是什么，以及为什么是这个目标。

实现模型和心理模型

  • 实现模型
    一种有关机器和程序实际运作方式的模型，其中会描述代码实现程序的细节。
    虽然可以充分体现工程上的基础结构，但没有为用户提供连贯机制，引导其来完成目标。最终，设计出的产品只会远离客户的需求，让用户感到迷惑。
  • 心理模型
    人们在使用产品时，不需要了解复杂机制工作原理的所有细节，因此可创造一种快捷认知的方式来解释复杂的机制。这种方式足以应付人们与产品的交互，但不一定能反映产品实际的内部工作机制。
  • 呈现模型（represented model）
    设计师选择如何向用户呈现程序的功能。
    呈现模型越趋向于用户的心理模型，用户就会感觉程序 / 产品越容易使用和理解。
    呈现模型越趋向于实际的实现模型，用户对应用软件的学习和使用能力就越低。

如果软件的呈现模型接近用户的心理模型，那就提供一套认知框架，使用户清楚地了解其目标和需求是如何实现的，从而消除用户界面中不必要的复杂度。

目标导向设计概论

  • 消除隔阂
    当前开发过程的主要问题之一是，大家各司其职，分工过度专业化。
    这种模式缺失了如何系统地把研究转化合成设计方案。

设计师最重要的工具：同理心（empathy）

  • 研究与蓝图之间：模型、需求和框架
    捕捉用户行为并且恰当地知道产品定义的方法并不多，大多数方法只是提供了一些任务层次上的信息，这些信息可以帮助确定界面布局、工作流程以及把功能转化为界面控件，但却不太擅长满足用户的广泛需求的基本框架。

于是，我们需要一个明确而系统的过程，弥合研究与设计的差距，定义用户模型，确定设计需求，并把这些内容转换为高层次的交互框架。——目标导向设计过程

  • 设计过程概述
    大致分为六个阶段：研究、建模、定义需求、定义框架、提炼和支持
    这些阶段遵循交互设计的五个组成部分：理解、抽象、架构、呈现和细节。

用户研究
运用观察和情景访谈，来获取有关产品的真正用户和潜在用户的定性数据。
实地观察和用户访谈的一个主要成果是，能够从中找出一组行为模式，帮助我们对现有或正在开发的产品的使用方式进行分类。这些模式指出了用户使用产品的目标和动机（即用户使用产品希望达到的具体或一般性结果）

用户建模
可将在分析实地研究和访谈结果中发现的行为和工作流模式，综合到领域和用户模型中。
在建模阶段，设计师采用多种方法工具来对人物模型综合、区分角色及其优先级，研究不同种类的用户目标，将人物映射到一系列行为中，保证不缺失或重复。

基于每个人物模型和其他人物模型的交叉程度，对不同目标进行对比，确定优先级，并通过这一过程，从一组人物模型中挑出具体的设计目标，选定人物角色类型的过程就决定了不同人物模型对产品最终设计形式和行为的影响程度。

需求定义
这一阶段主要采用场景为基础的设计方法，其中要突破在于，将满足具体人物模型的目标和需求置于首位，而不是关注抽象的用户任务。

一些分析通常会通过一种反复提炼的情景场景（context scenario）过程来完成。这一过程从人物模型在日常生活中使用产品开始，描述高层次的产品接触点，从而持续的一层一层深化下去，不断定义细节。

框架定义
在框架定义阶段，设计者创建整体的产品概念，为产品的行为、视觉设计，以及（如果适用的话）物理形态定义基本的框架。
交互设计团队采用两种重要方法论工具，加上情景场景，整合成了一套交互框架。

第一种是一组通用的交互设计原则，指导了在各种情境下，如何确定恰当的系统行为。
第二种重要的方法论工具是一组交互设计模式，对以前分析过问题中得出的通用解决方案（根据情景的不同可能会有所差异）进行分类。

数据和功能性需求在这种高度上进行描述后，设计师再按照交互原则将其变为设计元素，然后使用模式和原理组织为设计草图和行为描述。
这整个过程就是交互框架定义（Imeractkrn framework definition），即一种固定的设计概念，为后续设计细节提供了逻辑和高层次的形式结构。

产品成功的关键是目标，不是特性

成功的交互设计师始终会将其注意力放在用户目标上。
用户目标是实践交互设计的前提和基石。

目标导向设计是一个强有力的工具，能够回答定义和设计数字产品过程中出现的最重要的问题：

  • 用户是谁？
  • 用户在试图实现什么目标？
  • 用户如何看待他们要实现的目标？
  • 用户认为哪种体验具有吸引力并且值得的？
  • 产品应当如何工作？
  • 产品应当采用何种形式？
  • 用户如何与产品实现交互？
  • 产品功能如何能最有效地组织在一起？
  • 产品以何种方式面向首次使用的用户？
  • 产品如何在技术上实现易于理解、让人喜欢且易于操控？
  • 产品如何处理用户遇到的问题？
  • 产品如何帮助不常使用或者新手用户实现其目标？
  • 产品如何为专业级用户提供足够的深度和力度？
